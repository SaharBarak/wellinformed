# v2.1 Milestone Candidates — Decision Doc

**Drafted:** 2026-04-14 (overnight, autonomous)
**Status:** Awaiting user approval — pick A, B, C, D, or a combo

After v2.0 closed (6 phases, 313/313 tests, Wave 2 SOTA at 72.30% NDCG@10 SciFact, 4-wave benchmark story), here are the candidate v2.1 milestones. Each is sized for a 2-7 day milestone, scoped to one coherent theme, and gated by clear acceptance criteria.

---

## Candidate A — Ship Wave 2 to production *(highest user-facing impact)*

**Goal:** Migrate `src/infrastructure/embedders.ts` and `src/infrastructure/vector-index.ts` from the v1 baseline (MiniLM 384d, dense only, 64.82% NDCG@10) to the Wave 2 pipeline (nomic-embed-text-v1.5 768d + SQLite FTS5 BM25 RRF hybrid, 72.30% NDCG@10).

**End-user lift:** +7.48 NDCG@10 on retrieval quality. Real, measured, leaderboard-comparable.

**Scope (4 phases, ~5-7 days):**
- **Phase 21** — Foundation: extend `VectorIndex` port with `searchHybrid()`, `EmbedderConfig.model` field, error variant for dimension mismatch, `MigrationStatus` domain type. Embedder factory routes between `xenovaEmbedder({model: "all-MiniLM-L6-v2", dim: 384})` and `xenovaEmbedder({model: "nomic-ai/nomic-embed-text-v1.5", dim: 768})` based on config.
- **Phase 22** — FTS5 BM25 integration: add `fts_docs` virtual table to the production vec0 schema, write/read paths, BM25 sanitizer (port from `bench-beir-sota.mjs`), RRF fusion utility in `src/domain/vectors.ts`. Hybrid search becomes the default `searchGlobal` and `searchByRoom` path. Latency budget: 36 ms p50 on 5K corpus matches the benchmark.
- **Phase 23** — Migration: detect existing 384d vectors.db files on daemon start, prompt user to re-index, write a `wellinformed reindex --to nomic` CLI command that re-embeds and re-inserts existing nodes. Atomic: write to `vectors.db.new`, swap on success, never touch the old DB until the new one verifies. Bonus: keep MiniLM as an opt-in "fast mode" via `wellinformed config set encoder minilm` for users who don't want the 550 MB nomic download.
- **Phase 24** — TDD suite: cover new code paths (hybrid search, BM25 sanitizer, RRF fusion), migration paths (old DB detection, re-index, atomic swap), and a regression bar against Wave 2 numbers (must hit ≥72% NDCG@10 on the bundled SciFact mini-fixture). Update `bench-v2.sh` to use the new in-process pipeline so the bench numbers and the production numbers come from the same code.

**Acceptance criteria:**
1. `wellinformed ask "..."` returns hybrid (dense + BM25) top-K results by default
2. New install downloads nomic ONNX (~550 MB, one-time) and creates a 768d vectors.db
3. Existing 384d vectors.db files are detected and the user is offered re-index OR fast-mode opt-in
4. `npm test` includes a regression test asserting hybrid retrieval matches ≥72% NDCG@10 on a SciFact-derived fixture
5. README + docs/index.html updated to reflect production = Wave 2 (current docs already say "Wave 2 is measured but not yet shipped")

**Risks:**
- 550 MB ONNX download on first run will surprise users — needs a clear progress UI and a "skip / use minilm" escape hatch
- Existing user vectors.db migration is destructive without backup; the atomic-swap design must be airtight
- BM25 indexes add storage cost (FTS5 doubles ~2× the corpus column) — fine on local SSD but worth noting

**Why this is the obvious next milestone:** the +7.48 NDCG@10 lift is the largest single user-facing improvement available. It's measured, reproducible, and has a clear engineering path. The benchmark commit (`b893004`) literally documents this as the "deferred from v2.0" item.

## Candidate B — DHT internet-wide peer discovery *(P2P story completion)*

**Goal:** Turn on libp2p Kademlia DHT for internet-wide peer discovery. Currently DHT is wired but off-by-default (Phase 17 scoping decision). With DHT on, two wellinformed nodes anywhere on the public internet can find each other without manual `peer add`.

**Scope (3 phases, ~3-4 days):**
- **Phase 21** — DHT bootstrap + record publication: enable kad-dht, configure bootstrap nodes, publish own peer record on join
- **Phase 22** — Discovery integration: `wellinformed peer discover --dht` walks the DHT for peers advertising shared rooms, presents them with metadata for opt-in `peer add`
- **Phase 23** — TDD: 3-node DHT mesh test, peer record TTL, network partition recovery

**Acceptance:** two wellinformed instances on different home networks find each other via the public DHT within 60 seconds and can run federated search.

**Risks:**
- DHT pollution / spam — need rate limiting on advertised records
- Privacy concern — publishing your peer ID on the public DHT exposes you. Needs an opt-in flag with clear consent UX.
- Bootstrap node trust — picking the wrong bootstrap nodes can leak metadata to surveillance infrastructure

**Why this could be next:** completes the P2P story and moves wellinformed from "LAN demo" to "real internet-scale knowledge graph network." But it's smaller user-facing impact than Candidate A.

## Candidate C — LSP type-aware call graph *(code intelligence depth)*

**Goal:** Phase 19 used tree-sitter for syntactic parsing, leaving 70-80% of call edges marked `unresolved` on bigger codebases (because tree-sitter doesn't do type resolution). Replacing the resolver with LSP queries against running language servers (typescript-language-server, pyright, etc.) should turn most `unresolved` into `exact`.

**Scope (3 phases, ~4-5 days):**
- **Phase 21** — LSP client: minimal LSP protocol implementation, server lifecycle (spawn TS server per workspace, per Python venv, etc.), stdio framing
- **Phase 22** — Resolution: for each `unresolved` call edge from the syntactic pass, query LSP `textDocument/definition` to get the target symbol; upsert as `exact` confidence
- **Phase 23** — TDD: integration test against a fixture codebase with known type-resolved call graph, regression bar on `unresolved` count

**Acceptance:** wellinformed codebase index of itself shows `unresolved` count drop from ~4,500 to <500.

**Risks:**
- LSP server startup is slow (3-10s for typescript-language-server cold start) — async indexing path needed
- Language servers crash; need restart logic
- Some codebases have multiple language servers — picking the right one is non-trivial

**Why this could be next:** the most direct way to make `code_graph_query` more useful for agents. But it's the most complex of the four candidates.

## Candidate D — Web dashboard for sessions room *(UX completion)*

**Goal:** The Phase 13 web dashboard predates Phase 20 (sessions). Add a sessions view that lets users browse their indexed Claude Code sessions, search across them, and see assistant-message highlights per session.

**Scope (2 phases, ~2-3 days):**
- **Phase 21** — Backend: `dashboard.ts` adds `/sessions` route returning the sessions room with timeline + search + tool-call summary
- **Phase 22** — Frontend: vis.js timeline view of sessions, click to drill into a session, search bar, copy-as-context button

**Acceptance:** `wellinformed dashboard` shows a "Sessions" tab; clicking a session displays its messages with timestamps; search across sessions works.

**Risks:**
- Smallest scope, fewest risks. Mostly UI work.

**Why this could be next:** the lowest-effort visible win. Closes the loop between Phase 20 (auto-ingest) and Phase 13 (visual exploration).

## My recommendation

**Do A first** (Wave 2 production swap). Reasons:
1. **Largest user-facing impact** (+7.48 NDCG@10 is enormous for a memory tool — that's the difference between "matches BM25" and "competitive with bge-base").
2. **Deferred explicitly from v2.0** in the benchmark commit. The work is already measured; we just need to wire it into prod.
3. **Architecturally clean** — the bench scripts (`bench-beir-sota.mjs`) prove the pipeline works end-to-end. Phase 22 is mostly porting the bench logic into `src/infrastructure/vector-index.ts`.
4. **Honest follow-through** — the README currently says "Wave 2 is measured but not yet shipped." Shipping it closes that gap.

**Do D second** (sessions dashboard) if there's appetite for a quick visual win after a heavy infrastructure milestone.

**Defer B and C** to v2.2+. Both are valuable but neither is as immediately user-impactful as A.
